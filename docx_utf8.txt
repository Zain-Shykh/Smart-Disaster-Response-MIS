2. Normalization
This section walks through the normalization steps applied to the schema to eliminate data redundancy, prevent anomalies, and ensure data integrity. All tables have been brought to Third Normal Form (3NF), and selected tables satisfy Boyce-Codd Normal Form (BCNF) as well.

2.1  First Normal Form (1NF)
A table is in 1NF when every attribute holds only atomic (indivisible) values and each row is uniquely identifiable.

The following issues were identified from the initial conceptual model and corrected to achieve 1NF:

Composite attributes decomposed: Address (used in Citizen, DisasterEvent, RescueTeam, Warehouse, Donor) was decomposed into Street, Area, City, Province as individual columns. FullName was decomposed into FirstName and LastName across all applicable entities.
Multivalued attributes extracted: Phone numbers for User, Citizen, and Donor were moved into separate tables (UserPhone, CitizenPhone, DonorPhone) each with a composite primary key of the owner's ID and the phone number. Specializations for RescueTeam and Hospital were similarly moved to RescueTeamSpecialization and HospitalSpecialization.
Derived attributes separated: Attributes that are derivable from other data — TotalReports on Citizen and DisasterEvent, Duration on DisasterEvent and TeamActivity, OccupancyRate on Hospital, ResponseTimeMinutes and ResolutionTimeMinutes on EmergencyReport, LengthOfStayHours on PatientAdmission — are implemented as computed columns or maintained by triggers rather than stored redundantly.

After these corrections, every column in every table holds a single, indivisible value, and every row is uniquely identified by its primary key. All tables satisfy 1NF.

2.2  Second Normal Form (2NF)
A table is in 2NF when it is in 1NF and every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies). Partial dependencies can only occur in tables with composite primary keys.

Composite-key tables in this schema and their analysis:

UserRole (UserID, RoleID) → AssignedAt, AssignedBy: Both non-key attributes describe the assignment itself, not a single key part. Full dependency confirmed. No partial dependency.
RolePermission (RoleID, PermissionID): No non-key attributes. Trivially in 2NF.
TeamActivity (TeamID, ActivityID) → all activity attributes: ActivityID is a partial key that has no meaning without TeamID. All attributes describe the specific activity of a specific team. Full dependency confirmed.
InventoryAlert (InventoryID, AlertID) → all alert attributes: Same reasoning as TeamActivity. Full dependency confirmed.
ApprovalHistory (RequestID, HistoryID) → all history attributes: Full dependency confirmed.
UserPhone, CitizenPhone, DonorPhone (OwnerID, Phone): No non-key columns. Trivially in 2NF.
RescueTeamSpecialization, HospitalSpecialization (OwnerID, Specialization): No non-key columns. Trivially in 2NF.

All tables with composite keys were verified to have no partial dependencies. All tables satisfy 2NF.

2.3  Third Normal Form (3NF)
A table is in 3NF when it is in 2NF and every non-key attribute is non-transitively dependent on the primary key, i.e., no non-key attribute determines another non-key attribute.

Potential transitive dependencies examined:

Hospital: HospitalID → TotalBeds → OccupancyRate. OccupancyRate is computed from TotalBeds and AvailableBeds, both of which depend on HospitalID. Since OccupancyRate is a computed column (not stored independently), no transitive dependency violation exists.
Donation: DonationID → DonorID → donor name/contact. Donor attributes are stored in the Donor table, not repeated in Donation. Donation holds only the DonorID foreign key. No transitive dependency.
Expense: ExpenseID → EventID → disaster location. Event attributes are stored in DisasterEvent, not in Expense. No transitive dependency.
TeamAssignment: AssignmentID → AssignedBy → user details. User attributes are in the User table. No transitive dependency.
ResourceAllocation: AllocationID → InventoryID → (WarehouseID, ResourceID). Warehouse and resource details are in their own tables; ResourceAllocation holds only the foreign key. No transitive dependency.

After this analysis, no transitive dependencies remain in any table. All tables satisfy 3NF.

2.4  Boyce-Codd Normal Form (BCNF)
BCNF requires that for every non-trivial functional dependency X → Y, X must be a superkey. BCNF is a stricter form of 3NF and is typically only violated when a table has multiple overlapping candidate keys.

The Inventory table has a composite unique constraint on (WarehouseID, ResourceID), making it a candidate key alongside InventoryID. Since both candidate keys contain only key columns and no attribute determines another non-key attribute through a non-superkey determinant, no BCNF violation exists.

The Permission table has a composite unique constraint on (Module, Action, PermissionName). PermissionID is the primary key. Since (Module, Action, PermissionName) is also a candidate key, and no non-key attribute depends on only part of this, no BCNF violation exists.

All other tables have a single candidate key (their surrogate primary key), and therefore BCNF is automatically satisfied for those tables.


3. Design Rationale
This section justifies the major design decisions made in the database schema, from entity modeling to transaction handling, access control, automation, and performance considerations.

3.1  Entity and Relationship Decisions
Why 15 Strong Entities?
Each strong entity represents a domain object that has independent existence and its own lifecycle. For example, a Donor exists in the system even when no disaster is active. A Resource is catalogued before any disaster event. Treating these as independent entities avoids artificially constraining their management and allows the system to support pre-disaster preparedness.

Weak Entities
Three entities were modeled as weak because they are existentially dependent on a parent. TeamActivity only exists in the context of a RescueTeam and its activity identifier is only unique within a team. Similarly, ApprovalHistory exists only for a specific ApprovalRequest, and InventoryAlert is only meaningful in the context of a specific Inventory record. Using weak entities with composite keys correctly reflects this dependency and prevents orphaned records.

Associative Entities with Own Attributes
Several many-to-many relationships carry rich descriptive data that belongs to the relationship itself, not to either participating entity. For example, TeamAssignment records when the assignment happened, who authorized it, and what its current status is — none of which belong solely to a RescueTeam or to an EmergencyReport. Likewise, PatientAdmission records the patient's condition at admission, not a general property of the patient or the hospital. Modeling these as associative entities with their own attributes is the correct approach.

Location as Decomposed Composite Attribute
Location is a recurring concept appearing in Citizen, DisasterEvent, RescueTeam, Warehouse, and Donor. Rather than storing it as a single text field (which would be unsearchable) or introducing a separate Location table (which would add unnecessary joins for a simple address), each table contains the four components Street, Area, City, and Province as individual columns. Where geographic precision is needed — EmergencyReport, RescueTeam, Warehouse — Latitude and Longitude columns are added alongside the address components.

Multivalued Attributes as Separate Tables
Phone numbers and specializations are multivalued — an entity can have zero, one, or many of them. Storing them as a list in a single column would violate 1NF. They are therefore extracted to separate tables: UserPhone, CitizenPhone, DonorPhone, RescueTeamSpecialization, and HospitalSpecialization. Each of these uses a composite primary key of the parent's ID and the value itself, ensuring uniqueness without a surrogate key.

AuditLog as a Strong Entity
AuditLog was initially considered as a weak entity dependent on User. It was reclassified as a strong entity with a nullable UserID foreign key. This is necessary because audit records must survive even if a user account is deleted — the ON DELETE SET NULL rule is applied. Losing audit history when a user is removed would undermine compliance and forensic traceability, which are explicit system requirements.

3.2  Transaction Handling Approach
Several operations in the system involve multiple dependent steps that must succeed or fail as a unit to preserve data integrity. The following operations are implemented as explicit transactions:

Resource allocation: When a resource allocation is approved and dispatched, the Inventory Quantity is reduced and the ResourceAllocation status is updated in a single transaction. If either step fails, the entire operation rolls back, preventing inventory from being decremented without a confirmed allocation record.
Rescue team assignment: When a team is assigned to a report, the TeamAssignment record is inserted and the RescueTeam AvailabilityStatus is changed from Available to Assigned in one transaction. This prevents a team from appearing available while an assignment is being processed concurrently.
Financial approval: When an Expense or Donation record is updated, the ApprovalRequest status and ApprovalHistory entry are written in one transaction, ensuring that no approval decision can appear in the history without being reflected in the request status.

All transactions use the READ COMMITTED isolation level by default. For operations susceptible to race conditions — particularly concurrent allocation requests against the same inventory — serializable isolation or optimistic locking (checking the inventory quantity before committing) will be applied. Rollback logic is handled at the stored procedure level, with error messages logged to the AuditLog table.

3.3  RBAC Implementation Strategy
Role-Based Access Control is implemented through three tables: User, Role, Permission, bridged by UserRole and RolePermission. Roles represent functional positions in the disaster response organization: Administrator, Emergency Operator, Field Officer, Warehouse Manager, and Finance Officer. Permissions are defined at the module-action level, for example 'Reports:CREATE' or 'Finance:READ'.

This separation of roles from permissions (rather than hardcoding permissions into roles) allows fine-grained adjustment without schema changes. A new permission can be added to a role by inserting one row into RolePermission. The application layer checks permissions before every critical operation by joining UserRole and RolePermission on the current user's ID and the required permission name.

Password storage uses a hash (PasswordHash column), never plaintext. The AuditLog captures every login, data modification, and approval action, providing a complete trace of who did what and when.

3.4  Triggers and Database Automation
Database triggers are used to enforce business rules and propagate state changes automatically, reducing reliance on application-level logic and preventing inconsistencies when data is modified outside the main application.

Planned trigger implementations include:

After INSERT on TeamAssignment: Update RescueTeam.AvailabilityStatus to 'Assigned' and increment RescueTeam.TotalAssignments.
After UPDATE on TeamAssignment (Status = Completed): Update RescueTeam.AvailabilityStatus to 'Available'.
After INSERT/UPDATE on ResourceAllocation (Status = Dispatched): Deduct the allocated quantity from Inventory.Quantity and update Inventory.LastUpdated.
After UPDATE on Inventory.Quantity: Check whether Quantity falls below MinThreshold; if so, insert an InventoryAlert record with AlertType = 'LowStock' or 'OutOfStock'.
After INSERT on EmergencyReport: Increment DisasterEvent.TotalReports (if EventID is non-null) and Citizen.TotalReports.
After any data modification on sensitive tables: Insert a row into AuditLog capturing the action, table name, record ID, old value, new value, user ID, and timestamp.

Triggers ensure consistency even if the data is modified through direct SQL queries rather than the application. They also eliminate the need for the application to track cascading state changes manually, reducing the risk of partial updates.

3.5  Views for Abstraction and Security
Database views are used to serve two purposes: simplifying complex queries for dashboards and reports, and restricting data visibility per user role.

Planned views include:

ActiveEmergencyView: Joins EmergencyReport with DisasterEvent, Citizen, and TeamAssignment to show all active reports with their current assignment status. Used by Emergency Operators.
WarehouseInventoryView: Joins Inventory with Warehouse and Resource to show current stock levels, thresholds, and alert status. Used by Warehouse Managers.
FinancialSummaryView: Aggregates Donations and Expenses per DisasterEvent. Restricted to Finance Officers and Administrators.
HospitalCapacityView: Shows Hospital name, available beds, total beds, and occupancy rate. Used for hospital coordination.
TeamAvailabilityView: Shows RescueTeam with current status, location, and specializations. Used for dispatch decisions.
AuditReportView: Shows AuditLog entries joined with User names and table-readable descriptions. Restricted to Administrators.

Views that access financial or personal data will have column-level restrictions enforced by denying SELECT on base tables for non-admin roles and granting SELECT only on the relevant view.

3.6  Indexing Strategy and Performance
Given the high-volume, real-time nature of the system, query performance is a primary concern. The following indexing strategy has been designed based on expected query patterns:

Clustered indexes on primary keys are automatically created by SQL Server for all tables.
Non-clustered index on EmergencyReport.ReportTime: Emergency reports are queried frequently by time range, particularly for real-time dashboards. This index significantly reduces scan time for time-bounded queries.
Non-clustered index on EmergencyReport.DisasterType and EmergencyReport.SeverityLevel: Both columns are used for filtering and prioritization. A composite index on (DisasterType, SeverityLevel) supports queries that filter on both.
Non-clustered index on EmergencyReport.Status: Status-based filtering is extremely frequent (e.g., 'show all Pending reports'). A filtered index on Status = 'Pending' is considered to further optimize this common case.
Non-clustered index on DisasterEvent.Status and DisasterEvent.DisasterType: Supports event dashboard queries.
Non-clustered index on Inventory.WarehouseID and Inventory.ResourceID: Supports stock lookups by warehouse or resource type.
Non-clustered index on ResourceAllocation.EventID and ResourceAllocation.Status: Supports per-event allocation reports.
Non-clustered index on Donation.EventID and Donation.DonationDate: Supports financial summaries per event and per time period.
Non-clustered index on AuditLog.UserID and AuditLog.Timestamp: Supports audit trail queries by user and time period. Given the high write volume of AuditLog, the index overhead on inserts is accepted as a trade-off for query performance.

It is acknowledged that indexes add overhead to INSERT and UPDATE operations, which are frequent in high-volume scenarios such as emergency report submissions. The strategy therefore avoids over-indexing on insert-heavy tables and reserves indexes for columns that appear in WHERE clauses or JOIN conditions of frequent read queries. Actual query execution plans will be analyzed after the system is populated with representative data, and indexes will be adjusted accordingly.

-- Core user table with hashed credentials and soft-delete flag
CREATE TABLE [User] (
  UserID       INT           IDENTITY(1,1) PRIMARY KEY,
  FirstName    VARCHAR(100)  NOT NULL,
  LastName     VARCHAR(100)  NOT NULL,
  Username     VARCHAR(100)  NOT NULL UNIQUE,
  PasswordHash VARCHAR(255)  NOT NULL,
  Email        VARCHAR(255)  NOT NULL UNIQUE,
  CreatedAt    DATETIME2     NOT NULL DEFAULT SYSUTCDATETIME(),
  LastLoginAt  DATETIME2     NULL,
  IsActive     BIT           NOT NULL DEFAULT 1
);

-- Emergency reports submitted by citizens
CREATE TABLE EmergencyReport (
  ReportID     INT          IDENTITY(1,1) PRIMARY KEY,
  CitizenID    INT          NOT NULL REFERENCES Citizen(CitizenID),
  EventID      INT          NULL     REFERENCES DisasterEvent(EventID),
  City         VARCHAR(120) NOT NULL,
  Province     VARCHAR(120) NOT NULL,
  Latitude     DECIMAL(10,7) NULL CHECK (Latitude  BETWEEN -90  AND 90),
  Longitude    DECIMAL(10,7) NULL CHECK (Longitude BETWEEN -180 AND 180),
  DisasterType  VARCHAR(50)  NOT NULL,
  SeverityLevel VARCHAR(20)  NOT NULL
    CHECK (SeverityLevel IN ('Low','Medium','High','Critical')),
  ReportTime   DATETIME2    NOT NULL,
  Status       VARCHAR(20)  NOT NULL DEFAULT 'Pending'
    CHECK (Status IN ('Pending','InProgress','Resolved','Closed')),
  Source       VARCHAR(30)  NOT NULL
    CHECK (Source IN ('Mobile','Helpline','MonitoringSystem'))
);

-- Warehouse inventory with uniqueness and capacity guard
CREATE TABLE Inventory (
  InventoryID  INT            IDENTITY(1,1) PRIMARY KEY,
  WarehouseID  INT            NOT NULL REFERENCES Warehouse(WarehouseID),
  ResourceID   INT            NOT NULL REFERENCES Resource(ResourceID),
  Quantity     DECIMAL(14,2)  NOT NULL DEFAULT 0 CHECK (Quantity >= 0),
  MinThreshold DECIMAL(14,2)  NOT NULL CHECK (MinThreshold >= 0),
  MaxCapacity  DECIMAL(14,2)  NOT NULL CHECK (MaxCapacity >= 0),
  LastUpdated  DATETIME2      NOT NULL DEFAULT SYSUTCDATETIME(),
  CONSTRAINT UQ_Inventory_WR  UNIQUE (WarehouseID, ResourceID),
  CONSTRAINT CK_Qty_Cap       CHECK  (Quantity <= MaxCapacity)
);

-- Resource allocation with multi-step status lifecycle
CREATE TABLE ResourceAllocation (
  AllocationID INT           IDENTITY(1,1) PRIMARY KEY,
  InventoryID  INT           NOT NULL REFERENCES Inventory(InventoryID),
  EventID      INT           NOT NULL REFERENCES DisasterEvent(EventID),
  RequestedBy  INT           NOT NULL REFERENCES [User](UserID),
  Quantity     DECIMAL(14,2) NOT NULL CHECK (Quantity > 0),
  RequestTime  DATETIME2     NOT NULL,
  Status       VARCHAR(20)   NOT NULL
    CHECK (Status IN ('Pending','Approved','Dispatched','Consumed','Rejected')),
  DispatchedAt DATETIME2     NULL CHECK (DispatchedAt >= RequestTime)
);

-- Approval requests gate critical operations
CREATE TABLE ApprovalRequest (
  RequestID    INT          IDENTITY(1,1) PRIMARY KEY,
  RequestedBy  INT          NOT NULL REFERENCES [User](UserID),
  ReviewedBy   INT          NULL     REFERENCES [User](UserID),
  RequestType  VARCHAR(30)  NOT NULL
    CHECK (RequestType IN ('ResourceDistribution','RescueDeployment','Financial')),
  RequestTime  DATETIME2    NOT NULL,
  Status       VARCHAR(20)  NOT NULL DEFAULT 'Pending'
    CHECK (Status IN ('Pending','Approved','Rejected')),
  AllocationID INT NULL REFERENCES ResourceAllocation(AllocationID),
  AssignmentID INT NULL REFERENCES TeamAssignment(AssignmentID),
  ExpenseID    INT NULL REFERENCES Expense(ExpenseID)

4.3  Analytical Queries
The three queries below correspond directly to the MIS dashboards described in Section 11 and the reports controller endpoints (GET /api/Reports/...) implemented in the backend. Each query is described, followed by the SQL.

4.3.1  Incident Queue by City and Severity
Used by the Emergency Operator dashboard to show active reports grouped by city and severity. This query corresponds to the IncidentQueueByCitySeverity benchmark that was benchmarked WITH_INDEX and WITHOUT_INDEX (see Section 8).
SELECT er.City, er.SeverityLevel, COUNT(*) AS ReportCount,
       MIN(er.ReportTime) AS OldestReport
FROM EmergencyReport er
WHERE er.Status IN ('Pending', 'InProgress')
GROUP BY er.City, er.SeverityLevel
ORDER BY
  CASE er.SeverityLevel
    WHEN 'Critical' THEN 1 WHEN 'High' THEN 2
    WHEN 'Medium'   THEN 3 ELSE 4 END,
  ReportCount DESC;

4.3.2  Allocation History by Event and Status
Used by the Warehouse Manager dashboard to review resource dispatch and consumption per disaster event. This is the AllocationHistoryByEventStatus benchmark query and showed a 3 ms improvement (from 3 ms to 0 ms) after indexes were applied.
SELECT de.EventName, ra.Status,
       SUM(ra.Quantity) AS TotalQuantity,
       COUNT(ra.AllocationID) AS AllocationCount
FROM ResourceAllocation ra
JOIN DisasterEvent de ON ra.EventID = de.EventID
WHERE de.Status = 'Active'
GROUP BY de.EventID, de.EventName, ra.Status
ORDER BY de.EventName, ra.Status;

4.3.3  Financial Summary per Event
Used by the Finance Officer dashboard and the GET /api/Reports/financial/summary endpoint. The EventFinancialSummary benchmark measured near-zero latency in both indexed and non-indexed runs due to the small test dataset, confirming the view adds no measurable overhead on this query shape.
SELECT de.EventName, de.DisasterType,
  ISNULL(SUM(CASE WHEN don.Status='Confirmed' THEN don.Amount ELSE 0 END),0) AS TotalDonations,
  ISNULL(SUM(exp.Amount), 0) AS TotalExpenses,
  ISNULL(SUM(CASE WHEN don.Status='Confirmed' THEN don.Amount ELSE 0 END),0)
  - ISNULL(SUM(exp.Amount), 0)  AS NetBalance
FROM DisasterEvent de
LEFT JOIN Donation don ON don.EventID = de.EventID
LEFT JOIN Expense  exp ON exp.EventID = de.EventID
GROUP BY de.EventID, de.EventName, de.DisasterType
ORDER BY NetBalance ASC;


5.  Transaction Handling Demonstration
This section documents the three ACID transaction demonstrations executed as part of the database workstream (DB-08). All three demos were run successfully against the SQL Server test database using SQL_SCRIPTS/transaction_demos.sql and the results are recorded in the database latency benchmark report dated 23 April 2026.

5.1  Demo A: Resource Allocation Commit
Demo A validates that a resource allocation approval is atomic. The transaction inserts a ResourceAllocation record, creates a linked ApprovalRequest in Pending status, and decrements the Inventory quantity in one unit of work. The executed run produced the following committed outcome:
AllocationID = 4 (confirmed insert)
EventID = 2 (Quetta Earthquake 2026)
InventoryID = 1 (bottled water at Warehouse 1)
RequestID = 5 created with RequestType = ResourceDistribution
ApprovalStatus = Pending (correct initial state before approval decision)

The stored procedure pattern used is shown below. It uses BEGIN TRANSACTION / COMMIT / ROLLBACK with a TRY-CATCH block. The same pattern is enforced in the backend (ASP.NET Core 8) using Entity Framework Core’s transaction scope API, as validated by the integration test suite (54/54 passing).
BEGIN TRANSACTION;
BEGIN TRY
  -- Step 1: Insert allocation record
  INSERT INTO ResourceAllocation
    (InventoryID, EventID, RequestedBy, Quantity, RequestTime, Status)
  VALUES (1, 2, 1, 300, SYSUTCDATETIME(), 'Pending');

  DECLARE @AllocID INT = SCOPE_IDENTITY();

  -- Step 2: Create linked approval request
  INSERT INTO ApprovalRequest
    (RequestedBy, RequestType, RequestTime, Status, AllocationID)
  VALUES (1, 'ResourceDistribution', SYSUTCDATETIME(), 'Pending', @AllocID);

  -- Step 3: Reserve inventory (quantity check enforced by CHECK constraint)
  UPDATE Inventory
     SET Quantity = Quantity - 300, LastUpdated = SYSUTCDATETIME()
   WHERE InventoryID = 1 AND Quantity >= 300;

  IF @@ROWCOUNT = 0 THROW 50001, 'Insufficient stock', 1;
  COMMIT TRANSACTION;
END TRY
BEGIN CATCH
  ROLLBACK TRANSACTION;
  THROW;
END CATCH;

6.  Trigger Implementation and Use Cases
Database triggers were implemented in SQL_SCRIPTS/triggers.sql and validated using SQL_SCRIPTS/testing_Triggers.sql. All trigger behavior tests passed as part of DB-01 validation. Triggers use AFTER semantics so they fire post-commit within the same transaction, ensuring they roll back if the parent statement fails. The system defines six trigger groups covering the use cases below.

6.1  Team Assignment Status Trigger
Fires AFTER INSERT on TeamAssignment. Updates the assigned team’s AvailabilityStatus from Available to Assigned and increments TotalAssignments. This eliminates the need for the application layer to manage team state separately and ensures consistency even when assignments are inserted by direct SQL.
CREATE TRIGGER trg_TeamAssignment_AfterInsert
ON TeamAssignment AFTER INSERT AS
BEGIN
  SET NOCOUNT ON;
  UPDATE rt
     SET rt.AvailabilityStatus = 'Assigned',
         rt.TotalAssignments   = rt.TotalAssignments + 1
  FROM RescueTeam rt
  JOIN inserted   i  ON rt.TeamID = i.TeamID;
END;

-- Companion trigger: restore availability when assignment completes
CREATE TRIGGER trg_TeamAssignment_AfterUpdateComplete
ON TeamAssignment AFTER UPDATE AS
BEGIN
  SET NOCOUNT ON;
  UPDATE rt SET rt.AvailabilityStatus = 'Available'
  FROM RescueTeam rt
  JOIN inserted i ON rt.TeamID = i.TeamID
  WHERE i.Status = 'Completed';
END;

6.2  Inventory Alert Trigger
Fires AFTER UPDATE on Inventory. Detects the moment Quantity crosses below MinThreshold (not already below it) and inserts an InventoryAlert record with AlertType LowStock or OutOfStock. This ensures the Warehouse Manager is notified in real time without a scheduled batch job, and the alert is tied to the same transaction as the inventory update.
CREATE TRIGGER trg_Inventory_LowStockAlert
ON Inventory AFTER UPDATE AS
BEGIN
  SET NOCOUNT ON;
  -- Only fire when quantity crosses the threshold in this update
  INSERT INTO InventoryAlert (InventoryID, AlertType, AlertTime, Status)
  SELECT i.InventoryID,
         CASE WHEN i.Quantity = 0 THEN 'OutOfStock' ELSE 'LowStock' END,
         SYSUTCDATETIME(), 'Active'
  FROM inserted i
  JOIN deleted  d ON i.InventoryID = d.InventoryID
  WHERE i.Quantity  &lt; i.MinThreshold   -- now below threshold
    AND d.Quantity &gt;= d.MinThreshold;  -- was above threshold before update
END;

6.3  Emergency Report Count Trigger
Fires AFTER INSERT on EmergencyReport. Increments TotalReports on the Citizen record and, where the report is linked to a DisasterEvent, also increments TotalReports on that event. Both counters are maintained trigger-side to avoid application-layer aggregation queries on every dashboard load.
CREATE TRIGGER trg_EmergencyReport_AfterInsert
ON EmergencyReport AFTER INSERT AS
BEGIN
  SET NOCOUNT ON;
  -- Citizen counter
  UPDATE c SET c.TotalReports = c.TotalReports + 1
  FROM Citizen c JOIN inserted i ON c.CitizenID = i.CitizenID;

  -- Event counter (only where EventID is provided)
  UPDATE de SET de.TotalReports = de.TotalReports + 1
  FROM DisasterEvent de
  JOIN inserted i ON de.EventID = i.EventID
  WHERE i.EventID IS NOT NULL;
END;

6.4  Audit Log Trigger
A generic audit trigger is attached to all tables that handle sensitive data (EmergencyReport, ResourceAllocation, Donation, Expense, ApprovalRequest, and User). It fires AFTER INSERT, UPDATE, and DELETE, inserting one AuditLog row per operation with the action type, affected table, record ID, and before/after values serialised as JSON. This satisfies Requirements R12 (Audit and Monitoring) and R10 (Data Security).
CREATE TRIGGER trg_Expense_Audit
ON Expense AFTER INSERT, UPDATE, DELETE AS
BEGIN
  SET NOCOUNT ON;
  DECLARE @act NVARCHAR(10) =
    CASE
      WHEN EXISTS(SELECT 1 FROM inserted) AND EXISTS(SELECT 1 FROM deleted) THEN 'UPDATE'
      WHEN EXISTS(SELECT 1 FROM inserted) THEN 'INSERT'
      ELSE 'DELETE' END;

  INSERT INTO AuditLog (Action, TableName, RecordID, OldValue, NewValue, Timestamp)
  SELECT @act, 'Expense',
         ISNULL(CAST(i.ExpenseID AS VARCHAR), CAST(d.ExpenseID AS VARCHAR)),
         (SELECT * FROM deleted  d2 WHERE d2.ExpenseID = d.ExpenseID
          FOR JSON PATH, WITHOUT_ARRAY_WRAPPER),
         (SELECT * FROM inserted i2 WHERE i2.ExpenseID = i.ExpenseID
          FOR JSON PATH, WITHOUT_ARRAY_WRAPPER),
         SYSUTCDATETIME()
  FROM inserted i FULL OUTER JOIN deleted d ON i.ExpenseID = d.ExpenseID;
END;

6.5  Hospital Bed Management Trigger
Fires AFTER INSERT or UPDATE on PatientAdmission. When a patient is admitted (Status = Admitted) it decrements AvailableBeds on the linked Hospital. When a patient is discharged or transferred it increments AvailableBeds, keeping the hospital capacity view accurate for the auto-routing logic without requiring the application to perform separate bed updates.
CREATE TRIGGER trg_PatientAdmission_BedMgmt
ON PatientAdmission AFTER INSERT, UPDATE AS
BEGIN
  SET NOCOUNT ON;
  -- Decrement on new admission
  UPDATE h SET h.AvailableBeds = h.AvailableBeds - 1
  FROM Hospital h JOIN inserted i ON h.HospitalID = i.HospitalID
  WHERE i.Status = 'Admitted'
    AND NOT EXISTS (SELECT 1 FROM deleted d WHERE d.AdmissionID = i.AdmissionID);

  -- Increment on discharge or transfer
  UPDATE h SET h.AvailableBeds = h.AvailableBeds + 1
  FROM Hospital h JOIN inserted i ON h.HospitalID = i.HospitalID
  JOIN deleted   d ON d.AdmissionID = i.AdmissionID
  WHERE d.Status = 'Admitted'
    AND i.Status IN ('Discharged', 'Transferred');
END;


7.  View Definitions and Performance Comparison (Latency Analysis)
Database views were created in SQL_SCRIPTS/views.sql as part of DB-03. The latency comparison between views and direct base-table queries was executed and documented in SQL_SCRIPTS/performance_benchmarks.sql (DB-07) with results captured in the database latency benchmark report (DB-04 and DB-06). The views serve two purposes: simplifying complex joins for dashboard queries, and restricting column-level visibility per user role.

7.1  Defined Views

7.1.1  ActiveEmergencyView
Exposes active emergency reports with citizen name, event name, assigned team name, and assignment status. Used by the Emergency Operator dashboard and the GET /api/Reports/incidents/prioritized endpoint. Restricts visibility of internal CitizenID and TeamID foreign keys from read-only roles.
CREATE VIEW ActiveEmergencyView AS
SELECT
  er.ReportID, er.DisasterType, er.SeverityLevel, er.Status,
  er.City, er.Latitude, er.Longitude, er.ReportTime,
  c.FirstName + ' ' + c.LastName AS CitizenName,
  de.EventName,
  rt.TeamName, ta.Status AS AssignmentStatus
FROM EmergencyReport er
JOIN  Citizen         c  ON er.CitizenID = c.CitizenID
LEFT JOIN DisasterEvent  de ON er.EventID  = de.EventID
LEFT JOIN TeamAssignment ta ON ta.ReportID = er.ReportID
  AND ta.Status NOT IN ('Completed')
LEFT JOIN RescueTeam    rt ON ta.TeamID   = rt.TeamID
WHERE er.Status IN ('Pending', 'InProgress');

7.1.2  WarehouseInventoryView
Joins Inventory, Warehouse, and Resource to show current stock levels, thresholds, and a computed stock status label. Used by the Warehouse Manager dashboard and GET /api/Reports/resources/utilization. This view was included in the IncidentQueueByCitySeverity and AllocationHistoryByEventStatus benchmark runs.
CREATE VIEW WarehouseInventoryView AS
SELECT
  w.WarehouseName, w.City AS WarehouseCity,
  r.ResourceName, r.ResourceType, r.Unit,
  i.Quantity, i.MinThreshold, i.MaxCapacity,
  CASE
    WHEN i.Quantity = 0              THEN 'OutOfStock'
    WHEN i.Quantity &lt; i.MinThreshold THEN 'LowStock'
    ELSE 'Sufficient'
  END AS StockStatus,
  i.LastUpdated
FROM Inventory i
JOIN Warehouse w ON i.WarehouseID = w.WarehouseID
JOIN Resource  r ON i.ResourceID  = r.ResourceID;

7.1.3  FinancialSummaryView
Aggregates confirmed donations and expenses per DisasterEvent into a single net-balance row. Used by the Finance Officer dashboard and GET /api/Reports/financial/summary. Restricted to Finance Officer and Administrator roles at the application layer. This view was benchmarked as the EventFinancialSummary test case.
CREATE VIEW FinancialSummaryView AS
SELECT
  de.EventID, de.EventName, de.DisasterType,
  ISNULL(SUM(CASE WHEN don.Status = 'Confirmed' THEN don.Amount ELSE 0 END), 0)
    AS TotalDonations,
  ISNULL(SUM(exp.Amount), 0) AS TotalExpenses,
  ISNULL(SUM(CASE WHEN don.Status = 'Confirmed' THEN don.Amount ELSE 0 END), 0)
  - ISNULL(SUM(exp.Amount), 0) AS NetBalance
FROM DisasterEvent de
LEFT JOIN Donation don ON don.EventID = de.EventID
LEFT JOIN Expense  exp ON exp.EventID = de.EventID
GROUP BY de.EventID, de.EventName, de.DisasterType;

8.  Indexing Strategy and Query Performance Report
Custom indexes were created in SQL_SCRIPTS/indexes.sql (DB-05) and a dedicated performance benchmark script was added in SQL_SCRIPTS/performance_benchmarks.sql (DB-07). The explicit WITHOUT_INDEX versus WITH_INDEX comparison was documented in the database latency benchmark report (DB-06). This section describes the indexing rationale and observed results.

8.1  Indexes Created
The indexing strategy targets columns that appear in WHERE clauses or JOIN conditions of the most frequently executed dashboard and allocation queries. Both single-column and composite indexes are used, and a filtered index is applied for the most common dashboard filter.

-- Emergency reports: time-range dashboard (most recent reports first)
CREATE NONCLUSTERED INDEX IX_ER_ReportTime
  ON EmergencyReport (ReportTime DESC);

-- Emergency reports: filtered index for Pending status (highest-traffic filter)
CREATE NONCLUSTERED INDEX IX_ER_Pending
  ON EmergencyReport (ReportID, SeverityLevel)
  WHERE Status = 'Pending';

-- Emergency reports: composite filter on city and severity for incident queue
CREATE NONCLUSTERED INDEX IX_ER_City_Severity
  ON EmergencyReport (City, SeverityLevel)
  INCLUDE (Status, ReportTime, DisasterType);

-- Inventory: warehouse-resource lookups and stock checks
CREATE NONCLUSTERED INDEX IX_Inv_WarehouseResource
  ON Inventory (WarehouseID, ResourceID)
  INCLUDE (Quantity, MinThreshold, LastUpdated);

-- Resource allocation: per-event status aggregations
CREATE NONCLUSTERED INDEX IX_RA_Event_Status
  ON ResourceAllocation (EventID, Status)
  INCLUDE (Quantity, DispatchedAt);

-- Donation: per-event and date-range financial reports
CREATE NONCLUSTERED INDEX IX_Don_Event_Date
  ON Donation (EventID, DonationDate DESC);

-- Audit log: user-activity and time-range compliance queries
CREATE NONCLUSTERED INDEX IX_Audit_User_Time
  ON AuditLog (UserID, Timestamp DESC);

-- Disaster event: status-based filtering for active events
CREATE NONCLUSTERED INDEX IX_DE_Status
  ON DisasterEvent (Status)
  INCLUDE (EventName, DisasterType, StartTime);

8.2  Performance Observations
The benchmark results reported in Section 7.2 represent the actual executed output from the test database. The following summary aligns the observations with the indexing strategy and project requirements (R14, R19):
AllocationHistoryByEventStatus improved from 3 ms to 0 ms after IX_RA_Event_Status was applied. This is a direct benefit of the composite index eliminating a table scan on the ResourceAllocation table when filtering by EventID and Status together.
IncidentQueueByCitySeverity (BaseTable) improved from 4 ms to 0 ms after IX_ER_City_Severity was applied, confirming that the composite index on (City, SeverityLevel) with INCLUDE columns eliminates the key lookup.
EventFinancialSummary showed no measurable change because the test dataset is small. On a production-scale dataset with thousands of donation and expense rows, the IX_Don_Event_Date index is expected to reduce the sort and join cost meaningfully.
The IncidentQueueByCitySeverity View measured 9 ms versus 2 ms in the indexed run. As the benchmark report notes, this is attributable to execution plan variance on small datasets, not to index overhead.
Insert and update overhead: indexes on EmergencyReport, Inventory, and AuditLog add write overhead proportional to insert volume. The AuditLog table receives one write per sensitive mutation; the index overhead on it is accepted because the compliance query benefit outweighs it. For EmergencyReport, the filtered index on Status = Pending adds minimal insert overhead because the filter is narrow.

10.  Design Rationale Document
This section provides the comprehensive justification for the major design decisions made across the database schema, transaction handling architecture, RBAC model, trigger and view strategy, and indexing approach. It satisfies Requirements R17 and R19 from the project statement.

10.1  Entity and Relationship Decisions

10.1.1  Fifteen Strong Entities
Each strong entity represents a domain object that has independent existence and its own lifecycle. A Donor exists in the system before any disaster occurs. A Resource is catalogued in the warehouse before it is allocated. A Hospital maintains its bed records continuously. Treating these as independent strong entities avoids artificially constraining their management and allows the system to support pre-disaster preparedness, which is an explicit operational requirement.

10.1.2  Weak Entities
TeamActivity, ApprovalHistory, and InventoryAlert were modelled as weak entities because they are existentially dependent on a parent. A TeamActivity has no meaning outside the context of a specific RescueTeam, and its ActivityID is only unique within that team. Similarly, an ApprovalHistory entry exists only for a specific ApprovalRequest, and an InventoryAlert is only meaningful relative to a specific Inventory record. Using composite partial keys correctly reflects this dependency and prevents orphaned records if the parent is deleted. Note that AuditLog was initially considered weak but was reclassified as a strong entity with a nullable UserID (ON DELETE SET NULL) because audit records must survive user account deletion to preserve compliance traceability.

10.1.3  Associative Entities with Own Attributes
Several many-to-many relationships carry rich descriptive data that belongs to the relationship itself, not to either participating entity. TeamAssignment records when the assignment happened, who authorized it, and the progression from Assigned through Completed, none of which belong solely to a RescueTeam or to an EmergencyReport. Likewise, PatientAdmission records the patient’s clinical condition at the time of admission, not a general property of the patient or the hospital. Inventory records the quantity and thresholds for a specific resource at a specific warehouse, attributes that are properties of the stock relationship, not of either standalone entity.

10.1.4  Location as Decomposed Composite Attribute
Location is a recurring concept across Citizen, DisasterEvent, RescueTeam, Warehouse, Donor, and EmergencyReport. Rather than storing it as a single unstructured text field (which would be unsearchable) or extracting it to a shared Location table (which would add joins to every location-bearing entity), each table stores four atomic address columns (Street, Area, City, Province) directly. Where geographic precision is needed for distance-based routing, Latitude and Longitude decimal columns are added alongside the address components. This design supports the rescue team proximity ranking algorithm in the backend without requiring a spatial index.

10.2  Transaction Handling Approach
Three operations were identified as requiring explicit transactions because they involve multiple dependent table mutations that must succeed or fail as a unit. These are implemented at both the database layer (stored procedure BEGIN TRANSACTION / COMMIT / ROLLBACK with TRY-CATCH) and at the application layer (Entity Framework Core transaction scope on relational providers). The three protected operations are:
Resource allocation approval: updating ResourceAllocation status to Dispatched, decrementing Inventory.Quantity, and writing an AuditLog entry must be atomic. The CHECK constraint on Inventory.Quantity prevents the quantity from going negative, but the transaction ensures that no allocation record is marked Dispatched if the inventory decrement fails.
Rescue team assignment: inserting the TeamAssignment record and updating RescueTeam.AvailabilityStatus to Assigned must be atomic. The UPDLOCK hint in the stored procedure prevents two concurrent assignment requests from reading the same Available status and both succeeding.
Financial approval: updating an Expense or ApprovalRequest status and writing the ApprovalHistory entry in one transaction ensures that no approval decision can appear in the history without being reflected in the request status, which is critical for audit compliance.

The READ COMMITTED isolation level is the default. For high-contention allocation scenarios, the backend applies optimistic concurrency using a version token (RowVersion column), returning HTTP 409 Conflict on stale token updates and including the current token in the response for client-side retry. This was validated in the integration test suite.

10.3  RBAC Implementation Strategy
Role-Based Access Control is implemented through three tables (User, Role, Permission) bridged by UserRole and RolePermission. Roles represent functional positions: Administrator, EmergencyOperator, FieldOfficer, WarehouseManager, FinanceOfficer. Permissions are defined at the module-action level (e.g., Reports:READ, ResourceLogistics:UPDATE) allowing fine-grained adjustment without schema changes. A new permission is granted to a role by inserting one row into RolePermission; no code change is required.

At the application layer, every controller is decorated with [Authorize(Roles = ...)] class-level attributes. JWT tokens issued at login carry the user’s role claims. The PBKDF2-HMAC-SHA256 password hashing scheme (120,000 iterations, 16-byte random salt, 32-byte derived key) ensures that stored credentials are resistant to rainbow-table attacks and brute force. A non-breaking migration path allows legacy hashes to be transparently upgraded to PBKDF2 on first login.

10.4  Trigger Design Rationale
Triggers were chosen over application-layer state propagation for three reasons. First, they fire regardless of how data is modified, including direct SQL updates performed by administrators or data migration scripts. Second, they keep the application code thinner by removing cross-table consistency logic from the service layer. Third, they execute in the same transaction as the triggering statement, so they roll back automatically if the parent operation fails.

The deliberate use of AFTER triggers (rather than INSTEAD OF triggers) means that all constraint checks on the base table are satisfied before the trigger fires, preserving the CHECK constraint guarantees. The inventory alert trigger uses a before/after Quantity comparison (using the deleted pseudo-table) to fire only when the threshold is crossed in that specific update, not on every update where Quantity is already below threshold. This prevents duplicate alerts from being generated by repeated minor quantity adjustments.

10.5  View Strategy and Access Control
Views serve two distinct purposes in this system. For reporting, they pre-join frequently combined tables (EmergencyReport with Citizen and TeamAssignment, Inventory with Warehouse and Resource, DisasterEvent with Donation and Expense) so that the application layer can query a single view name rather than constructing the same multi-table JOIN in every query. For security, views restrict column-level visibility: FinancialSummaryView does not expose individual donation records or donor personal data, only aggregated totals per event. AuditReportView is restricted to the Administrator role entirely.

The benchmark results confirm that views add minimal overhead on small datasets (0-2 ms) and that any variance between view and base-table performance is attributable to execution plan differences on the small test dataset rather than inherent view overhead. For the FinancialSummaryView, converting to an indexed (materialized) view is noted as a future optimisation once the Donation and Expense tables exceed several thousand rows.

10.6  Indexing Trade-off Analysis
The indexing strategy deliberately avoids over-indexing insert-heavy tables. AuditLog receives one write per sensitive mutation across multiple tables; the IX_Audit_User_Time index adds write overhead proportional to mutation volume but is justified because compliance queries over the audit log are frequent and time-sensitive. EmergencyReport uses a filtered index on Status = Pending rather than a full-column index because Pending represents a small, rapidly-cycling subset of the total table. This minimizes both index size and insert overhead while providing the maximum benefit for the most common dashboard filter. The AllocationHistoryByEventStatus and IncidentQueueByCitySeverity benchmark results (3 ms and 4 ms improvements) provide empirical evidence that the chosen composite indexes directly address the query patterns generated by the backend reports controller.
